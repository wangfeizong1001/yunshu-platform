# 权限体系设计

本文档详细介绍云枢中台的权限控制体系，包括 RBAC 模型、数据权限、按钮级权限等。

---

## 一、权限模型概述

### 1.1 RBAC 模型

云枢中台采用 **RBAC**（Role-Based Access Control，基于角色的访问控制）模型：

```
┌────────┐      ┌────────┐      ┌────────────┐
│  User  │────▶│  Role  │────▶│ Permission │
│ (用户) │  n:n │ (角色) │  n:n │  (权限)    │
└────────┘      └────────┘      └────────────┘
     │              │
     │              └────────────┐
     ▼                           ▼
┌────────┐              ┌────────────┐
│  Dept  │              │ Menu/Button │
│ (部门) │              │  (菜单/按钮)│
└────────┘              └────────────┘
```

### 1.2 权限分层设计

```
┌─────────────────────────────────────────────────────────┐
│                    应用层权限                              │
│   ┌───────────────────────────────────────────────────┐  │
│   │     菜单权限（是否可见某个菜单/路由）              │  │
│   └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                    操作层权限                              │
│   ┌───────────────────────────────────────────────────┐  │
│   │   按钮权限（是否可见某个按钮/是否可调用接口）      │  │
│   └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                    接口层权限                              │
│   ┌───────────────────────────────────────────────────┐  │
│   │     API 权限（是否可调用某个 REST API）           │  │
│   └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                    数据层权限                              │
│   ┌───────────────────────────────────────────────────┐  │
│   │    数据权限（可见哪些数据/部门数据范围）           │  │
│   └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

## 二、菜单权限

### 2.1 菜单类型

| 类型 | 标识 | 说明 | 路由 |
|------|------|------|------|
| **目录** | M | 菜单分组，不对应实际页面 | /system |
| **菜单** | C | 具体页面，对应路由组件 | /system/user |
| **按钮** | F | 页面内的操作按钮 | N/A |

### 2.2 菜单树结构

```
系统管理 (M)
├── 用户管理 (C) ──▶ apps/admin/src/views/system/user/index.vue
│   ├── 用户查询 (F)
│   ├── 用户新增 (F)
│   ├── 用户修改 (F)
│   └── 用户删除 (F)
├── 角色管理 (C)
│   ├── 角色查询 (F)
│   ├── 角色新增 (F)
│   └── 角色分配权限 (F)
├── 菜单管理 (C)
└── 部门管理 (C)

系统监控 (M)
├── 操作日志 (C)
├── 在线用户 (C)
├── 服务监控 (C)
└── 定时任务 (C)
```

### 2.3 菜单加载流程

```
1. 用户登录成功
2. 前端请求 /getRouters 获取菜单树
3. 后端根据用户角色，过滤菜单表 sys_menu
4. 返回用户有权限的菜单列表（树形结构）
5. 前端动态生成路由 addRoute()
6. 菜单根据权限渲染
```

### 2.4 菜单相关 API

| API | 方法 | 说明 |
|-----|------|------|
| /getRouters | GET | 获取当前用户的菜单路由 |
| /system/menu/list | GET | 菜单列表（管理用） |
| /system/menu | POST | 新增菜单 |
| /system/menu/:id | PUT | 修改菜单 |
| /system/menu/:id | DELETE | 删除菜单 |

## 三、按钮级权限

### 3.1 权限标识

每个按钮对应一个**权限标识字符串**（perms）：

| 模块 | 操作 | 权限标识 |
|------|------|---------|
| 用户 | 查询 | `system:user:query` |
| 用户 | 新增 | `system:user:add` |
| 用户 | 修改 | `system:user:edit` |
| 用户 | 删除 | `system:user:remove` |
| 用户 | 导出 | `system:user:export` |
| 角色 | 分配权限 | `system:role:edit` |

### 3.2 前端实现

**v-permission 指令**

```vue
<!-- 仅 system:user:add 权限可见 -->
<el-button v-permission="'system:user:add'" @click="handleAdd">
  新增
</el-button>

<!-- 任一权限可见 -->
<el-button v-permission.any="['system:user:edit', 'system:user:add']">
  编辑
</el-button>

<!-- 编程式检查 -->
<script setup lang="ts">
import { hasPermission } from '@/utils/permission';

if (hasPermission('system:user:remove')) {
  // 显示删除按钮
}
</script>
```

**实现原理**

```typescript
// apps/admin/src/utils/permission.ts
// 核心逻辑:
// 1. 从 Pinia store 获取当前用户权限列表
// 2. 检查传入的权限标识是否在列表中
// 3. v-permission 指令控制元素显隐
```

### 3.3 后端实现

**基于注解的权限检查**

```java
// 伪代码（概念）
@PreAuthorize("@ss.hasPermi('system:user:list')")
@GetMapping("/list")
public List<User> list() {
    return userService.list();
}

@PreAuthorize("@ss.hasPermi('system:user:add')")
@PostMapping
public void add(@RequestBody User user) {
    userService.add(user);
}
```

## 四、数据权限

### 4.1 数据范围（Data Scope）

| 范围 | code | 说明 |
|------|------|------|
| **全部数据权限** | 1 | 可查看所有数据 |
| **自定义数据权限** | 2 | 自定义部门范围 |
| **本部门数据权限** | 3 | 仅本部门数据 |
| **本部门及以下** | 4 | 本部门及子部门 |
| **仅本人数据** | 5 | 仅本人创建的数据 |

### 4.2 数据权限实现原理

**查询自动注入条件**

```sql
-- 原始查询
SELECT * FROM sys_user WHERE status = 0;

-- 注入数据权限条件后（假设 data_scope = 4, 本部门及以下）
SELECT * FROM sys_user
WHERE status = 0
AND dept_id IN (
    SELECT dept_id FROM sys_dept
    WHERE dept_id = :currentDeptId
    OR find_in_set(:currentDeptId, ancestors)
);

-- 注入后（data_scope = 5, 仅本人）
SELECT * FROM sys_user
WHERE status = 0
AND create_by = :currentUserId;
```

### 4.3 数据权限优先级

多个角色时，采用**取最大范围**策略：

```
角色A: data_scope = 3 (本部门)
角色B: data_scope = 1 (全部数据)
→ 最终数据范围: 1 (全部数据)
```

## 五、角色管理

### 5.1 角色字段

| 字段 | 说明 | 示例 |
|------|------|------|
| role_name | 角色名称 | 普通用户 |
| role_key | 角色标识 | ROLE_USER |
| role_sort | 显示顺序 | 1 |
| data_scope | 数据范围 | 4 (本部门及以下) |
| dept_ids | 自定义部门 | [100, 101] 当 data_scope = 2 时 |
| menu_ids | 菜单权限 | [1, 2, 3, 100, 101] |
| status | 状态 | 0 正常 |

### 5.2 角色分配流程

```
1. 选择用户 → 分配角色
2. 选择角色 → 分配菜单权限
3. 选择角色 → 设置数据权限范围
4. 清除用户缓存（使其立即生效）
```

## 七、API 权限检查

### 7.1 权限检查中间件

```
请求到达
    │
    ▼
┌────────────────────┐
│ Token 校验          │
└────┬───────────────┘
     │ 有效
     ▼
┌────────────────────┐
│ 权限标识检查        │ (注解: hasPermi, hasRole)
└────┬───────────────┘
     │ 有权限
     ▼
┌────────────────────┐
│ 执行业务逻辑        │
└────────────────────┘
```

### 7.2 权限表达式

| 表达式 | 说明 |
|--------|------|
| `@ss.hasPermi('system:user:list')` | 检查权限标识 |
| `@ss.hasPermiOr(['system:user:list', 'system:user:add'])` | 任一权限 |
| `@ss.hasPermiAnd(['system:user:list', 'system:user:add'])` | 所有权限 |
| `@ss.hasRole('admin')` | 检查角色 |

## 八、权限缓存

### 8.1 缓存策略

| 缓存项 | 位置 | TTL | 失效时机 |
|--------|------|-----|---------|
| 用户信息 | Pinia + Redis | 2h | 登出 / 信息变更 |
| 权限列表 | Pinia + Redis | 2h | 角色变更 / 权限变更 |
| 菜单树 | Pinia + Redis | 1h | 菜单变更 |
| 字典数据 | LocalStorage + Redis | 1d | 后台刷新 |

### 8.2 权限变更通知

**触发缓存刷新的操作**:

1. 用户信息更新 → 清除 `user:{id}` 缓存
2. 角色权限变更 → 清除所有该角色用户的权限缓存
3. 菜单变更 → 清除菜单树缓存
4. 字典变更 → 清除对应字典缓存

**实现方式**:

```
事件驱动（Event Bus）:
- 用户更新事件 → 触发缓存清除
- 角色变更事件 → 触发缓存清除
- 菜单变更事件 → 触发缓存清除

定时轮询（兜底方案）:
- 前端每 30 分钟检查一次权限版本号
- 版本号变更则重新加载
```

## 九、多租户下的权限

### 9.1 租户内权限隔离

**每个租户有独立的**:
- 角色定义
- 用户角色分配
- 菜单权限配置

**平台级权限（跨租户）**:
- 平台管理员角色
- 平台级菜单
- 租户管理相关功能

### 9.2 权限检查优先级

```
┌──────────────────────────────────────────────┐
│  1. 是否为平台管理员?                         │
│     └─ 是 → 跳过租户数据隔离                 │
├──────────────────────────────────────────────┤
│  2. 检查 tenant_id 匹配                       │
│     └─ 所有查询自动注入 tenant_id 条件        │
├──────────────────────────────────────────────┤
│  3. 检查用户角色权限                          │
│     └─ 按标准 RBAC 流程                      │
└──────────────────────────────────────────────┘
```

## 十、安全建议

### 10.1 权限设计最佳实践

1. **最小权限原则**: 仅授予必要的权限
2. **角色分层**:
   - 超级管理员（系统级，谨慎使用）
   - 管理员（租户级）
   - 普通用户（部门级）
3. **权限审核**: 定期审计权限分配
4. **敏感操作二次确认**: 涉及删除、修改权限的操作需二次确认

### 10.2 常见漏洞防范

| 漏洞 | 防范措施 |
|------|---------|
| 越权访问 | 每个 API 都必须做权限检查，前端控制不可靠 |
| 水平越权 | 检查数据归属（确保用户只能操作自己/权限范围内的数据） |
| 垂直越权 | 严格校验 API 权限标识 |
| 权限泄露 | Token 安全存储，使用 HTTPS |
