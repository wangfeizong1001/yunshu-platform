# 多租户架构

本文档详细介绍云枢中台的多租户设计、数据隔离、套餐管理与租户生命周期管理。

---

## 一、租户模型

### 1.1 什么是租户（Tenant）

**租户**是系统中最大的组织单元，代表一个独立的企业/组织/客户。每个租户有独立的：

- 用户体系
- 角色权限
- 部门组织
- 菜单配置
- 业务数据
- 计费与套餐

### 1.2 租户在系统中的位置

```
┌──────────────────────────────────────────────────────┐
│                    平台层 (Platform)                   │
│  租户管理 | 套餐管理 | 系统监控 | 平台级配置            │
└──────┬───────────────────────────────────────────────┘
       │
┌──────┼──────────────┬───────────────────────────────┐
│      │              │                               │
┌─▼──────┐      ┌─▼──────┐                   ┌─▼──────┐
│ 租户A  │      │ 租户B  │    ...            │ 租户N  │
│ 企业X  │      │ 企业Y  │                   │ 企业Z  │
│ 100用户│      │ 20用户 │                   │ 500用户│
└──┬─────┘      └──┬─────┘                   └──┬─────┘
   │               │                             │
   └── 用户、角色、菜单、数据 ───────────────────────┘
```

### 1.3 平台与租户的区别

| 维度 | 平台 | 租户 |
|------|------|------|
| URL | `yunshu.example.com` | `tenant001.yunshu.example.com` |
| 用户体系 | 平台管理员 | 租户用户（企业员工） |
| 菜单 | 租户管理、套餐管理 | 系统管理、业务模块 |
| 数据可见性 | 所有租户数据汇总 | 仅本租户数据 |
| 登录入口 | 独立登录页 | 独立登录页（或统一登录后选择租户） |

## 二、数据隔离策略

### 2.1 隔离方案对比

| 方案 | 实现方式 | 隔离级别 | 性能 | 成本 | 适用场景 |
|------|---------|---------|------|------|---------|
| **逻辑隔离** | 所有租户共享表 + tenant_id 字段 | 逻辑 | 最佳 | 最低 | 中小企业、安全要求适中 |
| **Schema 隔离** | 每个租户独立 Schema | 中等 | 中等 | 中等 | 中型企业 |
| **物理隔离** | 每个租户独立数据库 | 最高 | 最差 | 最高 | 大型企业、安全要求极高 |

### 2.2 云枢中台采用的方案

**逻辑隔离（tenant_id）** + **Schema 隔离（可选）**

默认方案：**逻辑隔离**（所有表含 `tenant_id` 字段）

```sql
-- 核心表都包含 tenant_id 字段
CREATE TABLE sys_user (
    user_id     BIGINT PRIMARY KEY,
    tenant_id   VARCHAR(20) NOT NULL DEFAULT '000000',
    user_name   VARCHAR(30),
    -- ... 其他字段
    INDEX idx_tenant (tenant_id)
);

-- 典型查询（自动注入 tenant_id 条件）
SELECT * FROM sys_user
WHERE tenant_id = 'tenant001'  -- 自动注入
AND dept_id = 100;
```

### 2.3 数据自动过滤机制

**MyBatis DataPermissionInterceptor（概念）**

```
查询请求
    │
    ▼
┌────────────────────────────────┐
│ 判断是否需要租户隔离?             │
│ （检查注解 / 表名匹配规则）      │
└────┬───────────────────────────┘
     │ 是
     ▼
┌────────────────────────────────┐
│ 获取当前用户 tenant_id          │
│ （从 Token / Session 中获取）    │
└────┬───────────────────────────┘
     │
     ▼
┌────────────────────────────────┐
│ 注入 WHERE tenant_id = ? 条件   │
│ （不破坏原有 WHERE 条件）        │
└────┬───────────────────────────┘
     │
     ▼
执行查询
```

### 2.4 无需隔离的表

部分全局表（平台级配置）不需要 tenant_id 隔离：

| 表名 | 原因 |
|------|------|
| `sys_tenant` | 租户表本身是全局的 |
| `sys_package` | 套餐表全局配置 |
| `sys_config` | 全局参数配置 |
| `sys_dict_type / sys_dict_data` | 全局字典（可扩展租户级覆盖） |
| `gen_table / gen_table_column` | 代码生成配置 |

## 三、租户字段设计

### 3.1 租户表 (sys_tenant)

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | varchar(20) | 是 | 主键 |
| tenant_id | varchar(20) | 是 | 租户编号（如 T001） |
| contact_user_name | varchar(30) | 是 | 联系人 |
| contact_phone | varchar(30) | 是 | 联系电话 |
| company_name | varchar(30) | 是 | 企业名称 |
| license_number | varchar(30) | 否 | 统一信用代码 |
| address | varchar(200) | 否 | 地址 |
| logo | varchar(200) | 否 | Logo URL |
| intro | varchar(200) | 否 | 简介 |
| begin_time | datetime | 否 | 开始时间 |
| end_time | datetime | 否 | 结束时间 |
| status | char(1) | 是 | 状态（0 正常 1 停用） |
| del_flag | char(1) | 是 | 删除标志（0 存在 2 删除） |
| package_id | bigint | 是 | 套餐 ID |
| expire_time | datetime | 否 | 过期时间 |
| account_count | int | 否 | 账户数量（套餐配额） |
| user_count | int | 否 | 现有用户数 |
| create_time | datetime | 是 | 创建时间 |
| update_time | datetime | 否 | 更新时间 |

### 3.2 套餐表 (sys_package)

| 字段 | 类型 | 说明 |
|------|------|------|
| package_id | bigint | 套餐 ID |
| package_name | varchar(30) | 套餐名称（如基础版/企业版/旗舰版） |
| menu_ids | text | 关联菜单 ID 列表（JSON 数组格式） |
| status | char(1) | 状态 |
| remark | varchar(500) | 备注 |

### 3.3 套餐层级示例

| 套餐 | 价格 | 用户数限制 | 菜单权限范围 | 功能限制 |
|------|------|-----------|-------------|---------|
| **基础版** | ¥99/月 | 10 | 基础系统管理 + 报表 | 无工作流 |
| **企业版** | ¥299/月 | 50 | 全部系统管理 + 工作流 + 报表 | 无大屏看板 |
| **旗舰版** | ¥999/月 | 无限 | 全部功能 | 所有功能 |

## 四、用户认证流程

### 4.1 多租户登录流程

```
┌─────────────────────────────────────────────────────────┐
│           用户访问登录页                                  │
│   （通过子域名区分: tenant001.yunshu.example.com）        │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│ 1. 输入用户名/密码                                        │
│    （可选：支持手机号/邮箱登录）                           │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│ 2. 后端校验                                             │
│    ├─ 根据子域名解析 tenant_id                           │
│    ├─ 校验用户名/密码（在该租户下）                       │
│    ├─ 检查租户状态（是否已停用/过期）                     │
│    └─ 检查用户状态                                       │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│ 3. 生成 Token                                            │
│    ├─ JWT Payload:                                       │
│    │   { tenant_id, user_id, roles, exp }                │
│    └─ 写入 Redis （支持登出时失效）                        │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│ 4. 前端获取 Token                                        │
│    ├─ 存入 Pinia + LocalStorage                          │
│    └─ 后续请求自动携带 Authorization Header              │
└─────────────────────────────────────────────────────────┘
```

### 4.2 Token 结构（JWT Payload 示例）

```json
{
  "tenant_id": "tenant001",
  "user_id": 1001,
  "user_name": "zhangsan",
  "roles": ["ROLE_ADMIN", "ROLE_USER"],
  "permissions": ["system:user:list", "system:user:add"],
  "exp": 1705324800
}
```

### 4.3 租户状态校验

在登录和 Token 刷新时，必须校验：

1. **租户是否存在**
2. **租户状态是否正常**（`status = 0`）
3. **租户是否已过期**（`expire_time > now`）
4. **租户用户数是否超限**（`user_count <= account_count`）

## 五、菜单权限控制

### 5.1 菜单权限 = 角色权限 ∩ 套餐权限

```
套餐菜单: [1, 2, 3, 4, 5]
角色菜单:    [3, 4, 5, 6, 7]
                │
                ▼
实际可用菜单: [3, 4, 5]
```

**实现逻辑**：

```
用户请求菜单列表
    │
    ▼
┌─────────────────────────────────┐
│ 1. 查询用户角色对应的菜单权限     │
└────┬────────────────────────────┘
     │
     ▼
┌─────────────────────────────────┐
│ 2. 查询租户套餐对应的菜单权限     │
└────┬────────────────────────────┘
     │
     ▼
┌─────────────────────────────────┐
│ 3. 取交集（只保留两者都包含的）   │
└────┬────────────────────────────┘
     │
     ▼
返回菜单树
```

### 5.2 菜单过滤示例

| 菜单 | 套餐权限 | 角色权限 | 结果 |
|------|---------|---------|------|
| 用户管理 | ✅ | ✅ | ✅ 可见 |
| 角色管理 | ✅ | ✅ | ✅ 可见 |
| 工作流设计 | ✅ | ❌ | ❌ 不可见（角色无权限） |
| 大屏看板 | ❌ | ✅ | ❌ 不可见（套餐未购买） |
| 租户管理 | ❌ | ❌ | ❌ 不可见（平台级菜单） |

## 六、租户生命周期管理

### 6.1 租户状态流转

```
      注册/创建
  ┌───────────────┐
  ▼               │
┌──────────┐   停用    ┌──────────┐
│  正常    │────────▶ │  已停用   │
│ (status=0) │         │ (status=1) │
└────┬─────┘           └────┬─────┘
     │  启用                 │
     └───────────────────────┘
          │
          │ 续费
    ┌─────▼──────┐
    │  已过期    │
    │ (expired)  │
    └────────────┘
          │
          │ 超过保留期
    ┌─────▼──────┐
    │  已删除    │
    │ (软删除)   │
    └────────────┘
```

### 6.2 租户创建流程

```
平台管理员操作
    │
    ▼
┌─────────────────────────────────┐
│ 1. 创建租户记录                   │
│    sys_tenant 表                 │
└────┬────────────────────────────┘
     │
     ▼
┌─────────────────────────────────┐
│ 2. 创建租户管理员账号              │
│    sys_user (tenant_id = 新租户)  │
└────┬────────────────────────────┘
     │
     ▼
┌─────────────────────────────────┐
│ 3. 初始化租户基础数据              │
│    ├─ 初始化部门（总公司）         │
│    ├─ 初始化角色（管理员/普通用户）│
│    ├─ 初始化菜单（基于套餐）        │
│    └─ 初始化字典/配置             │
└────┬────────────────────────────┘
     │
     ▼
┌─────────────────────────────────┐
│ 4. 发送欢迎邮件/短信              │
│    包含登录信息 + 初始密码         │
└─────────────────────────────────┘
```

### 6.3 租户停用/启用

**停用操作**：

```
1. 设置 sys_tenant.status = 1
2. 清除该租户所有用户的 Token（Redis 中批量删除）
3. 记录操作日志
4. 通知该租户管理员
```

**启用操作**：

```
1. 恢复 sys_tenant.status = 0
2. 该租户用户重新登录即可使用
3. 记录操作日志
```

### 6.4 租户过期处理

```
每日定时任务（02:00）
    │
    ▼
┌─────────────────────────────────┐
│ 检查所有 expire_time < now 的租户 │
└────┬────────────────────────────┘
     │
     ├── 到期前 7 天 → 发送续费提醒
     │
     ├── 当天到期 → 发送最后提醒
     │
     └── 已过期 → 自动停用，数据保留 30 天
```

## 七、套餐切换

### 7.1 升级套餐（如 基础版 → 企业版）

```
1. 选择新套餐
2. 检查是否允许升级（用户数限制等）
3. 更新 sys_tenant.package_id
4. 刷新该租户所有用户的菜单权限缓存
5. 通知用户升级成功
```

### 7.2 降级套餐（如 企业版 → 基础版）

```
1. 检查降级后是否超出新套餐限制
   （如 当前用户数 > 新套餐用户数限制）
2. 更新 package_id
3. 刷新菜单权限缓存（交集变小）
4. 通知用户降级成功
```

## 八、数据备份与隔离

### 8.1 数据导出（租户自主）

```
租户管理员请求导出
    │
    ▼
┌─────────────────────────────────┐
│ 校验权限（仅租户管理员可操作）      │
└────┬────────────────────────────┘
     │
     ▼
┌─────────────────────────────────┐
│ 导出该租户所有数据为 SQL / JSON   │
│ （仅包含 tenant_id = 本租户的数据）│

     ▼

```

### 8.2 数据清理（平台级）

```
┌─────────────────────────────────┐
│ 策略: 软删除（del_flag）          │
│ 好处: 可恢复，有审计痕迹          │

     │
     ├── 超过 30 天 → 物理删除（可配置）
     │
     └── 重要操作记录到审计日志
```

## 九、性能优化

### 9.1 索引优化

```sql
-- 所有业务表必须在 tenant_id 上建立索引
CREATE INDEX idx_tenant ON sys_user(tenant_id);
CREATE INDEX idx_tenant ON sys_role(tenant_id);

-- 组合索引（tenant_id + 常用查询字段）
CREATE INDEX idx_tenant_user ON sys_user(tenant_id, user_name);
CREATE INDEX idx_tenant_time ON sys_oper_log(tenant_id, oper_time);
```

### 9.2 缓存策略

```
┌─────────────────────────────────┐
│ Key 规则: yunshu:{tenant_id}:{type}:{id} │
└─────────────────────────────────┘

示例:
- yunshu:tenant001:user:1001     (用户信息)
- yunshu:tenant001:menu:tree     (菜单树)
- yunshu:tenant001:role:list     (角色列表)
```

### 9.3 分库分表（可选）

```
按 tenant_id 哈希分片:
- tenant001 ~ tenant050 → DB1
- tenant051 ~ tenant100 → DB2
...

按时间归档（日志类）:
- sys_oper_log_202401
- sys_oper_log_202402
...
```
